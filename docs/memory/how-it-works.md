# How the Memory Layer Works

The memory layer is one of `claude-review`'s most powerful differentiators. It gives the tool a persistent, growing understanding of your codebase's recurring issues — something no single-pass review tool can offer.

## The problem it solves

Most AI code review tools treat every review in isolation. They don't know that:

- `src/db/users.go` has had a nil pointer bug fixed three times in the last month
- Your team consistently forgets to handle the error return from `os.ReadFile`
- The same SQL injection pattern keeps appearing in newly written query builders

`claude-review`'s memory layer solves this by storing findings across reviews and using them to make future reviews smarter.

## Architecture

The memory layer consists of three components:

### 1. SQLite database (`~/.claude-review/memory.db`)

Three tables:

| Table | Contents |
|-------|----------|
| `findings` | Every finding from every review: file, line, severity, description, hash |
| `consolidations` | Cross-PR pattern summaries generated by the consolidation agent |
| `false_positives` | Findings you've marked as false positives — excluded from future reviews |

The database lives at `.claude-review/memory.db` **in your repo root** — one database per repository. The `.claude-review/` directory is already included in the default `.gitignore`. It is local to your machine and never leaves it.

### 2. Query agent (before each review)

When you run `claude-review diff --memory`, before the finder agents are invoked:

1. The query agent looks up the files changed in this diff
2. It retrieves recent findings from those same files (last 30 days, capped at 20 findings per file)
3. It fetches the latest consolidation summaries
4. This context is formatted and injected into every finder agent's prompt

This means the finder agents know: *"In the last month, this file has had issues with X and Y — watch for them."*

### 3. Consolidation agent (background daemon)

The daemon runs a consolidation pass:

- **Time trigger**: every 30 minutes (if at least 1 new finding exists since the last run)
- **Volume trigger**: when 10+ new findings have been stored since the last consolidation

Either condition is sufficient. The first time the daemon runs with any findings stored, it consolidates immediately.

During consolidation, the agent receives only metadata (file paths, descriptions, severity, PR ref) — **no source code is ever sent**. It produces structured summaries of recurring patterns, which are stored in the `consolidations` table and surfaced by `claude-review insights`.

## Data flow

```
┌─────────────────────────────────────────────────────┐
│                  claude-review diff --memory          │
└──────────────────────────┬──────────────────────────┘
                           │
              ┌────────────▼────────────┐
              │     Query Agent          │
              │  Reads memory.db        │
              │  Fetches recent findings │
              │  for changed files      │
              └────────────┬────────────┘
                           │ context block
              ┌────────────▼────────────┐
              │   Finder Agents (×4–5)  │
              │   Verifier Agent        │
              │   Ranker Agent          │
              └────────────┬────────────┘
                           │ findings
              ┌────────────▼────────────┐
              │     Ingest Agent        │
              │  Stores new findings    │
              │  in memory.db           │
              └─────────────────────────┘
                           │ (async, background)
              ┌────────────▼────────────┐
              │  Consolidation Daemon   │
              │  Detects cross-PR       │
              │  patterns periodically  │
              └─────────────────────────┘
```

## What the memory context looks like

When memory is active, each finder agent receives a context block like this prepended to its prompt:

```
--- Memory Context ---
Files being reviewed that have past findings:

src/auth/token.go (3 findings in last 30 days):
  - [HIGH] JWT expiry not validated (2026-02-15)
  - [MEDIUM] Token signing key read from env without validation (2026-02-20)
  - [HIGH] Missing rate limiting on token generation endpoint (2026-03-01)

src/db/users.go (2 findings in last 30 days):
  - [MEDIUM] Nil pointer dereference before calling .Scan() (2026-02-10)
  - [HIGH] Nil pointer on empty result set not handled (2026-02-28)

Cross-PR pattern (from consolidation 2026-03-05):
  Authentication and authorization code has repeated nil-safety issues.
  Recommend a code review focused on error handling in the auth package.
--- End Memory Context ---
```

## Privacy

All data stays on your machine:

- The `memory.db` file is local to `~/.claude-review/`
- During consolidation, **only metadata** is sent to the Anthropic API (file paths, descriptions, severity counts) — **no source code**
- The memory database is not synced anywhere

## False positives

If a finding appears in memory that you consider a false positive (e.g., a pattern your codebase intentionally uses), you can mark it via the Go API (`memory.MarkRejected`) and it will be suppressed in future reviews. The threshold is **2 rejections** before a pattern is suppressed — this prevents accidentally silencing a real bug after a single dismissal.

Once a pattern hash has been rejected twice, `IsFalsePositive()` returns true and the ingest agent silently skips it on all future reviews.

A CLI command for managing false positives directly is planned for a future release. For now they can be managed through the SQLite DB:

```bash
sqlite3 .claude-review/memory.db \
  "SELECT * FROM false_positives WHERE repo = '<repo-hash>';"
```

## Clearing memory

```bash
claude-review memory clear
```

This wipes all findings, consolidations, and false positives. Useful if you want a fresh start after a major refactor.

## Database location

The default path is `<repo-root>/.claude-review/memory.db` — each repo has its own database. The daemon's PID file and log live separately in `~/.claude-review/`.

You can override the DB path in your config:

```json
{
  "memory_db": "/path/to/custom/memory.db"
}
```
